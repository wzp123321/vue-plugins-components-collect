# Git 提交记录规范

## 4.1 格式化 commit message 的目的

- 提供更多的历史信息，方便快速浏览
- 可过滤某些提交（如文档改动），便于快速查找信息
- 可直接从提交记录自动生成 Change log

## 4.2 commit message 的格式

任意一行不得超过 72 个字符（推荐 72，上限不超过 100），避免自动换行影响可读性。

基本结构（scope 为当前开发分支名，必填）：

```
<type>(<scope>): <subject>

<body>

<footer>
```

- 第一行 header 必填，`<type>` 必填，`<scope>` 必填（必须等于当前开发分支名），`<subject>` 必填
- body、footer 可省略

### 提交类型 type

- feat：功能变更或新增
- fix：缺陷修复
- docs：文档或注释变更
- style：代码风格调整（格式化、空格、分号等），不影响逻辑
- refactor：代码重构（非新功能、非修复）
- perf：性能优化
- test：单元测试或自动化相关变更
- build：影响构建系统或外部依赖的变更
- ci：持续集成相关变更（如 CI 配置、流水线脚本）
- chore：其他杂项变更（不影响 src 与 test）
- revert：回滚提交

### 范围 scope（必须为当前开发分支名）

- 取值为当前开发分支名，如：`feature/user-export`、`bugfix/auth-401`、`hotfix/v1.2.3`
- 分支命名建议：`feature/xxx`、`bugfix/xxx`、`hotfix/xxx`、`chore/xxx`、`release/x.y.z`
- 若使用短分支名，保持小写、kebab-case，简短明确

### 主题 subject

- 使用命令式、现在时：描述“做什么”，而非“已经做了什么”
- 不要大写首字母
- 结尾不加句号
- 简洁清晰，控制在 72 字符以内

示例（scope 为当前分支名）：

```
feat(feature/user-export): 新增用户导出功能
```

### 正文 body（可选）

- 使用命令式、现在时
- 说明动机、变更点、与之前行为的对比
- 可使用项目符号分点描述

示例：

```
实现基于后端导出接口的前端触发与状态提示
- 新增导出按钮与进度反馈
- 增加失败重试与取消能力
```

### 尾部 footer（可选）

- BREAKING CHANGE：声明非兼容性改动、原因、迁移方式
- 关闭问题：使用 `Closes #123` 或 `Closes #123, #456`

示例：

```
BREAKING CHANGE: 移除旧版导出入口，请改用 settings -> export
Closes #123
```

## 4.3 禅道问题单提交规范

在 footer 或单独行中追加以下约定，支持多编号逗号分隔：

- 单个 BUG

```
bug #BUG单号 描述信息
```

- 多个 BUG

```
bug #BUG单号1,BUG单号2 描述信息
```

- 单个 任务

```
task #TASK编号 描述信息
```

- 多个 任务

```
task #TASK编号1,TASK编号2 描述信息
```

示例完整提交：

```
feat(user): 新增导出用户数据

增加导出按钮、进度反馈与失败重试能力

Closes #456
task #1024 优化导出流程
```

## 4.4 快速模板

建议使用以下模板进行提交信息编写：

```
<type>(<scope>): <subject>

<body 可选，说明动机、改动点、对比、影响>

<footer 可选：BREAKING CHANGE: ...>
<footer 可选：Closes #123, #456>
<footer 可选：bug #BUG1,BUG2 描述>
<footer 可选：task #TASK1,TASK2 描述>
```

## 4.5 检查清单

- [ ] type 是否正确、规范
- [ ] subject 是否为命令式、现在时，长度不超过 72
- [ ] 是否需要 scope，并保持简短明确
- [ ] 是否存在 BREAKING CHANGE 并说明迁移方式
- [ ] 是否正确关联问题（Closes/bug/task）
